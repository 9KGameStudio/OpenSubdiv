00001 // CPU Subdivision with OpenSubdiv 
00002 // -------------------------------
00003 // In this example program, we will setup an OpenGL application that uses OSD to
00004 // subdivide an animated mesh. It is intended to be as simple as possible and 
00005 // not necessarily efficient. It is also intended as a learning tool for 
00006 // understanding the OSD internals. Unlike the other OSD examples, the common 
00007 // code infrastructure has been removed for clarity.
00008 //
00009 // ### Program Structure 
00010 //
00011 // This example program is structured as follows:
00012 //
00013 // 1. Setup static mesh topology (OsdHbrMesh)
00014 // 2. Convert the topology into a subdividable mesh (OsdMesh)
00015 // 3. On each frame: 
00016 //      * Animate the coarse mesh points and update the OsdMesh
00017 //      * Subdivide the updated mesh
00018 //      * Draw the subdivided mesh and wire frame
00019 //
00020 // If you are completely new to OSD, you should read the following sections to 
00021 // get a basic understanding of how it works.
00022 //
00023 // ### OSD Architecture Basics
00024 // As a client, you will primarily be interacting with the Osd and Hbr classes, 
00025 // however it's good to be aware of all three layers. The following describes
00026 // these layers from lowest level (Hbr) to highest (Osd):
00027 //
00028 // **Hbr: Halfedge Boundary Representation.**
00029 // This layer represents the mesh topology as meshes, vertices and edges. It is
00030 // the core that provides the structure for subdivision and provides an 
00031 // abstraction for dealing with topology in a type-agnostic way (i.e. everything
00032 // is templated).
00033 //
00034 // **Far: Feature Adaptive Representation.** 
00035 // Far uses hbr to create and cache fast run time data structures for table 
00036 // driven subdivision. Feature-adaptive refinement logic is used to adaptively 
00037 // refine coarse topology only as much as needed. The FarMesh does hold vertex 
00038 // objects but the topology has been baked into FarSubdivisionTables. It also
00039 // provides the underpinnings for generic dispatch of subdivision evaluation, so
00040 // subdivision can be preformed with different mechanisms (GLSL, Cuda, etc.),
00041 // the concrete implementations are specified at the next layer up.
00042 //
00043 // **Osd: Open Subdiv.**
00044 // Osd contains client level code that uses Far to create concrete instances of 
00045 // meshes and compute patch CVs with different back ends for table driven 
00046 // subdivision. Currently, the following are supported in Osd:
00047 //
00048 //  * CPU / C++ with single or multiple threads 
00049 //  * GLSL kernels with transform feedback into VBOs 
00050 //  * OpenCL kernels 
00051 //  * CUDA kernels
00052 //
00053 // The amount of hardware specific computation code is small, ~300 lines of code,
00054 // so it isn't a large effort to support multiple different ones for different 
00055 // clients. In the future, it is conceivable that additional dispatchers will be
00056 // developed to target mobile devices.
00057 //

00419 // ### Update Geometry and Subdivide
00420 
00421 // This is where the magic happens. Given the initial subdivision table stored 
00422 // in the OsdMesh, on every frame we can now send coarse point position updates 
00423 // and recompute the subdivided surface based on the coarse animation.
00424 //
00425 void
00426 updateGeom() {
00427     int nverts = (int)g_orgPositions.size() / 3;
00428 
00429     std::vector<float> vertex;
00430     vertex.reserve(nverts*6);
00431 
00432     const float *p = &g_orgPositions[0];
00433     const float *n = &g_normals[0];
00434 
00435     //
00436     // Apply a simple deformer to the coarse mesh. We save the deformed points 
00437     // and normals into a separate buffer to avoid accumulation of error. This 
00438     // loop really has nothing to do with OSD.
00439     // 
00440     float r = sin(g_frame*0.001f);
00441     for (int i = 0; i < nverts; ++i) {
00442         float move = 0.05f*cosf(p[0]*20+g_frame*0.01f);
00443         float ct = cos(p[2] * r);
00444         float st = sin(p[2] * r);
00445         
00446         vertex.push_back(p[0]*ct + p[1]*st);
00447         vertex.push_back(-p[0]*st + p[1]*ct);
00448         vertex.push_back(p[2]);
00449 
00450         //
00451         // To be completely accurate, we should deform the normals here too, but
00452         // the original undeformed normals are sufficient for this example 
00453         //
00454         vertex.push_back(n[0]);
00455         vertex.push_back(n[1]);
00456         vertex.push_back(n[2]);
00457  
00458         p += 3;
00459         n += 3;
00460     }
00461 
00462     //
00463     // Send the animated coarse positions and normals to the vertex buffer.
00464     //
00465     g_vertexBuffer->UpdateData(&vertex[0], nverts);
00466 
00467     //
00468     // Dispatch subdivision work based on the coarse vertex buffer. At this 
00469     // point, the assigned dispatcher will queue up work, potentially in many
00470     // worker threads. If the subdivided data is required for further processing
00471     // a call to Synchronize() will allow you to block until the worker threads
00472     // complete.
00473     //
00474     g_osdmesh->Subdivide(g_vertexBuffer, NULL);
00475 
00476     //
00477     // The call to Synchronize() is not actually necessary, it's being used

00505                     * p2 = &pos[f->GetVertex(2)->GetID()*3];

00508         cross( n, p0, p1, p2 );
00509 
00510         for (int j = 0; j < f->GetNumVertices(); j++) {
00511             int idx = f->GetVertex(j)->GetID() * 3;
00512             result[idx  ] += n[0];
00513             result[idx+1] += n[1];
00514             result[idx+2] += n[2];
00515         }
00516     }
00517     for (int i = 0; i < nverts; ++i)
00518         normalize(&result[i*3]);
00519 }
